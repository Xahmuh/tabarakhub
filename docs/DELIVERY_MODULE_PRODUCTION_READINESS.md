## Data Model Review

Primary delivery order table:

```text
public.delivery_orders
```

Production-supported fields now:

| Field | Current behavior |
|---|---|
| `id` | UUID primary key generated by Supabase. |
| `branch_id` | Required FK to `public.branches`. Branch users record only their own branch through RLS. |
| `order_date` | Required date. Branch users are constrained by RLS to today/yesterday inserts and same-day deletes. |
| `value_bhd` | Required numeric value, greater than zero. Service validation rejects non-finite or non-positive values. |
| `payment_type` | Required. Supported values: `BP`, `CARD`, `CASH`, `TALABAT`. |
| `pharmacist_id` | Optional FK. Recording UI loads only pharmacists assigned to the branch. Service validation rejects a pharmacist not assigned to the order branch. |
| `pharmacist_name` | Snapshot from the validated active pharmacist when `pharmacist_id` is present. |
| `driver_id` | Optional FK to `delivery_drivers`. Service validation rejects inactive/unavailable drivers. |
| `block_number` | Required for non-Talabat orders. Trimmed by the service. Talabat orders force this to null. |
| `area_name` / `governorate` | Database trigger snapshots from `delivery_blocks`; not trusted from the browser payload. |
| `is_outside_governorate` | Database trigger derives this from branch classification and resolved block governorate. |
| `notes` | Optional trimmed text. |
| `created_by`, `updated_by`, timestamps | Populated by the app/session and database defaults. |

Unknown/unresolved block behavior:

- Missing block is allowed only for `TALABAT`.
- Non-Talabat orders require a block number.
- If a branch enters a block number that is not in `delivery_blocks`, the order can be saved through the explicit "Save anyway" flow.
- That order is classified as unresolved: `block_number` exists but `area_name` and `governorate` are null.
- Coverage analytics separates missing/no-block orders from unresolved block-number orders.

Future-only fields not present on `delivery_orders`:

- Delivery status.
- Promised/delivered/SLA timestamps.
- Customer identifier.
- Product/category/order-item fields.

Because those fields do not exist, SLA, customer, repeat-rate, product, and category analytics remain future-only.

## Validation Hardening

Current hardening:

- `DeliveryHub` now exposes the recording form only to real branch users with delivery edit permission. This prevents manager/admin identity rows from being used as invalid `branch_id` values.
- `BranchRecordingPage` validates date and positive BHD value before saving.
- `deliveryService.orders.insert` validates branch, date, value, payment type, and non-Talabat block number.
- `deliveryService.orders.insert` trims block numbers and notes, forces Talabat block to null, and does not accept area/governorate from the client.
- `deliveryService.orders.insert` verifies selected pharmacists are active and assigned to the order branch.
- `deliveryService.orders.insert/update` rejects inactive/unavailable delivery drivers.
- `deliveryService.orders.update` validates dates, values, payment type, and trims block/notes before sending updates.
- LocalStorage fallback is not used for delivery orders in production.

Remaining validation items for real deployment:

- Confirm Supabase policies reject direct browser attempts to create delivery orders for another branch.
- Confirm database constraints reject invalid payment type, missing non-Talabat block, and non-positive value.
- Confirm pharmacist assignment RLS/read policies are acceptable for the client.

## Role/RLS Review

Current repo role model:

- `manager`: full delivery management in the current model; can manage delivery reference data, create operations tasks, and view all branches allowed by RLS.
- `owner`: read-broad role; delivery analytics/profitability visible read-only in the UI where RLS allows.
- `supervisor`: read-only delivery access, scoped to assigned branches by `supervisor_branches` and `current_app_is_supervisor_of`.
- `branch`: records and views own branch delivery data only.
- `warehouse`: current replacement for legacy admin/accounts; delivery defaults to none unless explicitly configured.
- Legacy `admin` and `accounts`: older role helpers supported these names, but the current role restructure migrates them away. Any still-present legacy profiles must be validated explicitly.
- `anon`: no delivery table grants.

RLS surfaces to validate on a real Supabase project:

- `delivery_orders` select/insert/update/delete.
- `delivery_blocks`, `delivery_areas`, `delivery_drivers`, `delivery_supervisors`, `branch_classifications`, and `delivery_cost_settings`.
- `operations_tasks` and `operations_task_events` when created from delivery coverage insights.

See `docs/DELIVERY_RLS_MANUAL_TESTS.md`.

## Coverage Analytics Review

Current coverage logic:

- Defaults to a bounded last-30-days window.
- Fetches date-scoped real `delivery_orders` only.
- Paginates order reads and caps at 50,000 rows to avoid silent truncation loops.
- Treats Talabat as non-mappable/no-block by design.
- Separates:
  - known block orders,
  - unknown/missing block orders,
  - unresolved block-number orders.
- Uses known-block orders as the denominator for per-block share.
- Uses mappable orders as the denominator for unknown-block rate and branch footprint.
- Gates trend signals below a 4-order sample as `insufficient_data`.
- Suppresses confident campaign advice below 10 mappable orders or when unknown-block rate is 40% or higher.
- Words expansion as further review only, never "open a branch".
- Disables SLA/customer/product analytics because required fields do not exist.
- Uses guarded denominators to avoid NaN/division-by-zero display risk.

Remaining validation:

- Confirm analytics are RLS-scoped with real manager, owner, supervisor, branch, and anon sessions.
- Confirm high-volume ranges remain acceptable in the target project.

## Bahrain Block Map Review

Current map behavior:

- Loads `public/data/bahrain-blocks.geojson` lazily through `fetch`.
- Does not bundle the 3.23 MB GeoJSON into JavaScript.
- Fails closed into an unavailable dataset if missing, non-200, invalid, or schema-unrecognized.
- Keeps matrix fallback available.
- Uses no paid map API and no map key.
- Renders only real polygons from the dataset; no geometry is invented.
- Shows mapped served block count, unmapped served block count, and total geometry block count.
- Keeps licensing/commercial redistribution notes in docs, not as scary in-app copy.
- `.gitignore` excludes `bahrain-geojson-main/`; optimized public assets remain tracked.

```text
External/commercial redistribution license not confirmed.
```

If the app is packaged or sold externally, replace the dataset with an approved licensed source or obtain written permission.

## Operations Task Integration Review

Current behavior:

- Coverage insights can be converted to tasks only through explicit button clicks.
- There is no auto-create behavior.
- UI checks for an open/in-progress task for the same insight and warns before creating another.
- Service uses `source_module = delivery_coverage`.
- `related_record_type` is stable: `delivery_block`, `branch_coverage`, or `delivery_insight`.
- `related_record_id` is stable: block number, branch id, or insight id.
- Supabase errors are shown to the user.
- Database RLS remains the enforcement boundary for task creation.

Current role caveat:

- The current role restructure makes `manager` the write-management role. Older deployments may still allow legacy `admin` through `current_app_can_manage`; validate the target project before documenting admin write access.
